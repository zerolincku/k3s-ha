# 架构说明

## 拓扑

本方案在 K3s 上部署 6 个 Pod：

```text
redis-0      Redis 初始 master
redis-1      Redis replica
redis-2      Redis replica
sentinel-0   Redis Sentinel
sentinel-1   Redis Sentinel
sentinel-2   Redis Sentinel
```

Redis Sentinel 模式是单主多副本，不是多主写入。任意时刻只有一个 Redis master 接收写请求。

## 服务入口

集群内暴露两个 Service：

```text
redis-headless.redis.svc.cluster.local
redis-sentinel.redis.svc.cluster.local:26379
```

`redis-headless` 用于 StatefulSet 稳定 DNS，不建议业务直接把它当写入口使用。

业务客户端应连接 Sentinel：

```text
Sentinel 地址: redis-sentinel.redis.svc.cluster.local:26379
Master 名称: mymaster
Redis 密码: 来自 redis-auth Secret
```

客户端通过 Sentinel 查询当前 master，再连接真正的 Redis master。

## 高可用机制

- 3 个 Sentinel 使用 quorum=2，避免单个 Sentinel 误判直接触发故障转移。
- Redis Pod 使用 StatefulSet，Pod 名称稳定。
- Redis 和 Sentinel 都设置了跨节点反亲和，尽量分散到 3 台 K3s master。
- 如果 K3s master 有 `NoSchedule` 控制面 taint，manifest 已包含 control-plane/master toleration。
- PodDisruptionBudget 要求 Redis 和 Sentinel 自愿驱逐时至少保留 2 个可用 Pod。

## 不持久化边界

本方案明确关闭 Redis 持久化：

```text
appendonly no
save ""
```

数据目录使用 `emptyDir`。Pod 重建、节点重启、故障转移过程中数据可能丢失或回退。这个方案适合：

- 缓存
- 可重建的会话
- 临时限流计数
- 测试环境

不适合：

- 订单、账务、库存等不可丢失数据
- 需要 RPO=0 的业务
- 长期保存的队列数据

## 重要限制

Kubernetes Service 不能自动识别 Redis 当前 master。本方案不提供固定的 `redis-master` 写入口 Service。业务必须使用支持 Sentinel 的 Redis 客户端。

如果业务客户端不支持 Sentinel，应改用 Redis Cluster、带代理的 Redis 方案，或引入专门的 Redis Operator。
