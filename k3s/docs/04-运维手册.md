# K3s HA 运维手册

## 日常巡检

```bash
kubectl get nodes -o wide
kubectl get nodes -L kubernetes.io/arch,node.k3s-ha.io/cgroup-version
kubectl -n kube-system get pods -o wide
kubectl get componentstatuses 2>/dev/null || true
kubectl get --raw=/readyz?verbose
```

在每台 master 上检查：

```bash
systemctl is-active k3s
ss -lntp | grep ':6443'
test "$(stat -fc %T /sys/fs/cgroup)" = "cgroup2fs" && echo cgroup-v2 || echo cgroup-v1
```

## API 入口检查

K3s 部署完成后，kubeconfig 默认可以先指向 master1 的 `6443`。K3s 侧只检查当前 kubeconfig 是否能访问 apiserver：

```bash
kubectl get --raw=/readyz
```

如果后续部署了 VIP/HAProxy，VIP、VRRP、HAProxy 后端检查放在 [../../keepalived-haproxy/docs/04-运维手册.md](../../keepalived-haproxy/docs/04-运维手册.md)。

## 系统组件镜像拉取失败

如果 `kube-system` Pod 长时间处于 `ContainerCreating`、`ErrImagePull` 或 `ImagePullBackOff`：

```bash
kubectl -n kube-system get pods -o wide
kubectl -n kube-system describe pod <pod-name>
```

重点看事件中的镜像地址和错误。如果出现下面这类信息，通常是节点到 Docker Hub 的网络问题：

```text
registry-1.docker.io
auth.docker.io
EOF
tls: handshake failure
```

处理建议：

1. 生产环境优先切换为 K3s airgap 包，避免依赖公网 Docker Hub。
2. 如果使用内网 registry，确认已经同步 `rancher/mirrored-pause`、`rancher/mirrored-coredns-coredns`、`rancher/mirrored-metrics-server`、`rancher/local-path-provisioner`、`rancher/mirrored-library-busybox` 等 K3s 系统镜像。
3. 如果只替换 pause 镜像，CoreDNS、metrics-server、local-path-provisioner 仍可能继续失败。
4. 临时排查时可以在节点上执行 `sudo k3s crictl pull <image>` 或 `sudo k3s ctr -n k8s.io image pull <image>` 验证 containerd 是否能拉镜像。

## Etcd Snapshot

建议每天至少一次 snapshot，并将 snapshot 复制到集群外部存储。

手动备份：

```bash
sudo k3s etcd-snapshot save --name manual-$(date +%Y%m%d%H%M%S)
sudo k3s etcd-snapshot ls
```

默认本地目录：

```text
/var/lib/rancher/k3s/server/db/snapshots
```

## 单节点故障

如果一台 master 宕机：

```bash
kubectl get nodes
kubectl get --raw=/readyz?verbose
```

预期还有 2 台 master Ready，控制面仍可用。修复故障节点后，先检查：

```bash
systemctl status k3s
journalctl -u k3s -n 200 --no-pager
```

如果节点数据损坏，不要直接删除 etcd 成员。先在健康 master 上确认成员状态，再按 K3s 官方恢复流程处理。

## 两节点故障

两台 master 同时故障时，etcd quorum 丢失，控制面不可用。处理顺序：

1. 优先恢复原有 master，不要同时重装多台。
2. 至少恢复 2 个 etcd 成员后再处理业务层故障。
3. 如果必须从 snapshot 恢复，先完整备份当前三台机器的 `/var/lib/rancher/k3s`。

## 证书与 kubeconfig

部署脚本生成的 kubeconfig 默认指向 `KUBECONFIG_SERVER`；如果配置里留空，则指向：

```text
https://<MASTER1_HOST>:6443
```

如果后续增加 VIP/HAProxy 并希望 kubeconfig 走统一入口，需要：

1. 确认三台 master 的 K3s `tls-san` 包含 VIP 或域名；如果部署前已设置 `TLS_SAN_VALUES`，可跳过。
2. 必要时重新启动 K3s，使证书包含新的 SAN。
3. 更新 kubeconfig server 地址。
4. 更新 `keepalived-haproxy/` 或外部 LB 配置。

## 生产变更原则

- 不在三台 master 上同时重启 K3s。
- 不在未备份 etcd snapshot 前重装或清理 `/var/lib/rancher/k3s`。
- 不使用未固定版本的在线安装命令升级生产集群。
- 优先在测试环境验证 K3s 小版本升级。
- 不在未验证业务镜像 multi-arch 支持前混跑 `amd64` 与 `arm64` 工作负载。
- 不随意修改 `K3S_CGROUP_DRIVER`；修改前先确认 kubelet 与 containerd 使用同一套 cgroup driver 策略。
