# 运维手册

## 常用检查

Pod 状态：

```bash
kubectl -n redis get pods -o wide
kubectl -n redis get statefulset
kubectl -n redis get pdb
```

Sentinel 视角：

```bash
kubectl -n redis exec sentinel-0 -- redis-cli -p 26379 sentinel masters
kubectl -n redis exec sentinel-0 -- redis-cli -p 26379 sentinel replicas mymaster
kubectl -n redis exec sentinel-0 -- redis-cli -p 26379 sentinel sentinels mymaster
```

Redis 角色：

```bash
password=$(kubectl -n redis get secret redis-auth -o jsonpath='{.data.password}' | base64 -d)

for pod in redis-0 redis-1 redis-2; do
  echo "===== $pod ====="
  kubectl -n redis exec "$pod" -- \
    redis-cli -a "$password" --no-auth-warning role
done
```

## 故障转移验证

删除当前 master Pod：

```bash
kubectl -n redis exec sentinel-0 -- \
  redis-cli -p 26379 sentinel get-master-addr-by-name mymaster

kubectl -n redis delete pod <当前-master-pod>
```

等待 Sentinel 选出新 master：

```bash
watch -n 2 'kubectl -n redis exec sentinel-0 -- redis-cli -p 26379 sentinel get-master-addr-by-name mymaster'
```

故障转移刚触发后，短时间内可能看到旧 master 和新 master 同时报告 `master`。这属于 Sentinel 重新配置旧 master 的收敛窗口，不应在这个窗口内判断故障转移失败。

持续检查三台 Redis 角色，直到只剩一个 master：

```bash
for i in $(seq 1 30); do
  masters=0
  for pod in redis-0 redis-1 redis-2; do
    role=$(kubectl -n redis exec "$pod" -- \
      sh -c 'redis-cli -a "$REDIS_PASSWORD" --no-auth-warning role | head -n1')
    echo "$pod $role"
    [ "$role" = master ] && masters=$((masters + 1))
  done
  [ "$masters" -eq 1 ] && break
  sleep 5
done
```

## 数据风险

本方案没有持久化。以下操作都可能导致数据丢失：

- 删除 Redis Pod。
- 节点重启。
- Redis 容器重建。
- 故障转移后旧 master 重新加入。

因此不要把不可丢失数据放到这个集群。

## 扩缩容边界

本 manifest 固定 3 个 Redis Pod 和 3 个 Sentinel Pod。不要直接把 Redis StatefulSet 扩到更多副本后就认为自动高可用完成；Sentinel 拓扑、反亲和、客户端连接和故障转移策略都要重新验证。

## 卸载

```bash
kubectl delete namespace redis
```

如果修改了 namespace：

```bash
kubectl delete namespace <REDIS_NAMESPACE>
```
