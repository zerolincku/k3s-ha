# 离线部署方案

离线部署分两段：在有网络的机器准备离线包，在离线环境执行同一个部署脚本。

## 离线包内容

离线包包含：

```text
k3s
install.sh
k3s-airgap-images-amd64.tar.zst
sha256sum-amd64.txt
metadata.env
```

`arm64` 离线包中的文件名对应为：

```text
k3s
install.sh
k3s-airgap-images-arm64.tar.zst
sha256sum-arm64.txt
metadata.env
```

说明：离线包内部统一把 K3s binary 放成 `k3s`，但下载时 `amd64` 使用 release 里的 `k3s`，`arm64` 使用 release 里的 `k3s-arm64`。

如果目标系统启用 SELinux，还需要单独准备：

```text
k3s-selinux-*.rpm
container-selinux
policycoreutils
selinux-policy
```

OS 包 `curl`、`socat`、`conntrack-tools` 等不包含在 K3s 官方离线包内，需要提前通过内网 yum/apt 源或 OS 安装包仓库解决。Keepalived/HAProxy 的离线依赖由 `keepalived-haproxy/` 目录单独描述。

## 固定版本

本方案固定使用已经验证过的版本：

```text
K3S_VERSION=v1.35.5+k3s1
```

如需升级版本，先在测试环境验证 3 master、CoreDNS、metrics-server、local-path-provisioner 和业务入口，再修改 `k3s/config.example.env` 与生产配置。

## 下载 K3s 本体资源

如果目标机器不能访问 `https://get.k3s.io` 或 GitHub release，需要先在有网络的机器下载 K3s 本体资源：

```bash
cp k3s/config.example.env k3s/prod.env
vim k3s/prod.env

K3S_ARCH=all bash k3s/scripts/download-k3s-assets.sh k3s/prod.env
```

输出目录：

```text
k3s/artifacts/assets/v1.35.5+k3s1/amd64/k3s
k3s/artifacts/assets/v1.35.5+k3s1/amd64/install.sh
k3s/artifacts/assets/v1.35.5+k3s1/amd64/sha256sum-amd64.txt
k3s/artifacts/assets/v1.35.5+k3s1/arm64/k3s-arm64
k3s/artifacts/assets/v1.35.5+k3s1/arm64/install.sh
k3s/artifacts/assets/v1.35.5+k3s1/arm64/sha256sum-arm64.txt
```

`download-k3s-assets.sh` 会校验 K3s binary 的 checksum。`install.sh` 来自 `https://get.k3s.io`，需要和当前 K3s 安装方式一起归档保存。

## 只下载系统镜像

如果目标机器可以在线安装 K3s binary，但不能稳定访问 Docker Hub，可以只在有网络的机器下载 K3s 官方 airgap 镜像归档：

```bash
cp k3s/config.example.env k3s/prod.env
vim k3s/prod.env

K3S_ARCH=all bash k3s/scripts/download-k3s-images.sh k3s/prod.env
```

输出目录：

```text
k3s/artifacts/images/v1.35.5+k3s1/amd64/k3s-airgap-images-amd64.tar.zst
k3s/artifacts/images/v1.35.5+k3s1/arm64/k3s-airgap-images-arm64.tar.zst
k3s/artifacts/images/v1.35.5+k3s1/<arch>/k3s-images.txt
```

`k3s-images.txt` 是本版本 K3s 系统镜像清单，可用于同步到内网 registry 或做审计。

部署时把归档路径写入配置或通过环境变量传入：

```bash
K3S_IMAGE_TAR_AMD64=/path/to/k3s-airgap-images-amd64.tar.zst \
K3S_IMAGE_TAR_ARM64=/path/to/k3s-airgap-images-arm64.tar.zst \
bash k3s/scripts/deploy-k3s-ha.sh k3s/prod.env
```

部署脚本会按节点架构选择对应归档，并复制到每台节点：

```text
/var/lib/rancher/k3s/agent/images/
```

使用 K3s 内置 containerd 时，K3s 启动会自动导入这个目录下的镜像归档。脚本还会创建：

```text
/var/lib/rancher/k3s/agent/images/.cache.json
```

用于条件导入，避免 K3s 重启时重复导入未变化的归档。这个方式只解决镜像问题，不包含 K3s binary、install.sh 和 OS 依赖。

## 散文件离线部署

如果已经分别下载了 K3s 本体资源和系统镜像归档，可以不制作 bundle，直接把散文件传给部署脚本：

```bash
K3S_AIRGAP=true \
K3S_BINARY_AMD64=/path/to/assets/v1.35.5+k3s1/amd64/k3s \
K3S_BINARY_ARM64=/path/to/assets/v1.35.5+k3s1/arm64/k3s-arm64 \
K3S_INSTALL_SCRIPT=/path/to/assets/v1.35.5+k3s1/amd64/install.sh \
K3S_IMAGE_TAR_AMD64=/path/to/images/v1.35.5+k3s1/amd64/k3s-airgap-images-amd64.tar.zst \
K3S_IMAGE_TAR_ARM64=/path/to/images/v1.35.5+k3s1/arm64/k3s-airgap-images-arm64.tar.zst \
bash k3s/scripts/deploy-k3s-ha.sh k3s/prod.env
```

脚本会在每台节点上完成：

1. 按节点架构选择 `K3S_BINARY_<ARCH>`，安装到 `/usr/local/bin/k3s`。
2. 安装 `K3S_INSTALL_SCRIPT` 到 `/opt/k3s-airgap/install.sh`。
3. 按节点架构选择 `K3S_IMAGE_TAR_<ARCH>`，复制到 `/var/lib/rancher/k3s/agent/images/`。
4. 创建 `/var/lib/rancher/k3s/agent/images/.cache.json`。
5. 使用 `INSTALL_K3S_SKIP_DOWNLOAD=true` 执行本地安装。

## 准备完整离线包

在有网络的机器执行：

```bash
cp k3s/config.example.env k3s/prod.env
vim k3s/prod.env

bash k3s/scripts/prepare-airgap-bundle.sh k3s/prod.env
```

默认输出：

```text
k3s/artifacts/airgap/k3s-airgap-bundle-v1.35.5+k3s1-amd64.tar.gz
```

可以通过环境变量指定版本和架构：

```bash
K3S_VERSION=v1.35.5+k3s1 \
K3S_ARCH=amd64 \
bash k3s/scripts/prepare-airgap-bundle.sh k3s/prod.env
```

命令行环境变量会覆盖 `k3s/prod.env` 中的同名配置，适合临时生成某个版本或某个架构的离线包。

同时准备 `amd64` 与 `arm64`：

```bash
K3S_VERSION=v1.35.5+k3s1 \
K3S_ARCH=all \
bash k3s/scripts/prepare-airgap-bundle.sh k3s/prod.env
```

输出示例：

```text
k3s/artifacts/airgap/k3s-airgap-bundle-v1.35.5+k3s1-amd64.tar.gz
k3s/artifacts/airgap/k3s-airgap-bundle-v1.35.5+k3s1-arm64.tar.gz
```

## 离线部署

把离线包复制到运维机，然后执行：

```bash
K3S_AIRGAP=true \
AIRGAP_BUNDLE_AMD64=/path/to/k3s-airgap-bundle-v1.35.5+k3s1-amd64.tar.gz \
AIRGAP_BUNDLE_ARM64=/path/to/k3s-airgap-bundle-v1.35.5+k3s1-arm64.tar.gz \
bash k3s/scripts/deploy-k3s-ha.sh k3s/prod.env
```

即使 `k3s/prod.env` 中保留了 `K3S_AIRGAP=false`，上面的命令行环境变量也会覆盖它。

如果三台机器都是同一架构，也可以只设置：

```bash
AIRGAP_BUNDLE=/path/to/k3s-airgap-bundle-v1.35.5+k3s1-amd64.tar.gz
```

脚本会将离线包分发到每台 master，并在每台机器上：

1. 按节点架构选择对应离线包。
2. 解压 `k3s` 到 `/usr/local/bin/k3s`。
3. 解压 `install.sh` 到 `/opt/k3s-airgap/install.sh`。
4. 复制 `k3s-airgap-images-<arch>.tar.zst` 到 `/var/lib/rancher/k3s/agent/images/`。
5. 创建 `.cache.json`，避免 K3s 重启时重复导入未变化的镜像归档。
6. 使用 `INSTALL_K3S_SKIP_DOWNLOAD=true` 执行本地安装脚本。

## 离线镜像策略

小规模 3 master 集群有两种选择：

| 方式 | 适用场景 | 说明 |
| --- | --- | --- |
| 每节点放置 airgap tarball | 最简单，适合 3 台机器 | 本方案默认使用 |
| 内网私有 registry | 后续服务很多，镜像治理要求高 | 需要维护 `registries.yaml` 和镜像同步流程 |

后续 Redis、RabbitMQ、MinIO 等服务建议统一进入内网 registry。K3s 自身可继续用 airgap tarball，也可切换为 registry 方式。

混合架构离线集群强烈建议使用支持 multi-arch manifest 的内网 registry。单独导入某个架构镜像 tarball 只能保证该节点本地可用，不能保证业务 Pod 在另一种架构节点上也能拉起。

如果在线部署时发现 `kube-system` Pod 卡在 `ImagePullBackOff`，并且事件中出现 `registry-1.docker.io`、`auth.docker.io`、`EOF` 或 `tls: handshake failure`，不要继续依赖公网重试。应改用本章的 airgap 包，或先把 K3s 系统镜像完整同步到内网 registry。

## 升级流程

1. 在联网机器下载新版本 K3s binary、install.sh、airgap images。
2. 生成新的离线包并校验 checksum。
3. 先备份 etcd snapshot。
4. 逐台 master 执行升级，保证任意时刻至少 2 台 master 健康。
5. 验证 kube-system Pod、节点版本、业务入口。

升级前备份：

```bash
sudo k3s etcd-snapshot save --name pre-upgrade-$(date +%Y%m%d%H%M%S)
```
