# 架构说明

Rook Ceph 在 K3s 中把 Ceph 作为 Kubernetes 原生存储集群运行。Rook 负责部署和调谐 Ceph 组件，Ceph 负责真正的数据存储。

本方案同时启用两种能力：

```text
CephFS
  文件系统能力，提供 ReadWriteMany PVC。

RGW
  对象存储网关，提供 S3 兼容 API。
```

## 3 master 拓扑

```text
k3s-master-1          k3s-master-2          k3s-master-3
-------------         -------------         -------------
k3s server            k3s server            k3s server
etcd                  etcd                  etcd

Ceph MON              Ceph MON              Ceph MON
Ceph OSD              Ceph OSD              Ceph OSD
独立数据盘             独立数据盘             独立数据盘

Ceph MGR active       Ceph MGR standby
Ceph MDS active       Ceph MDS standby
RGW Pod               RGW Pod
```

业务侧看到的是：

```text
Pod -> PVC(ReadWriteMany) -> StorageClass rook-cephfs -> CephFS
业务 -> S3 endpoint       -> Ceph RGW                 -> Ceph OSD
```

## 核心组件

```text
Rook Operator
  管理 Ceph 集群生命周期。

Ceph MON
  集群仲裁和状态维护。3 节点部署 3 个 MON。

Ceph MGR
  管理组件，提供 Dashboard、metrics 和管理接口。

Ceph OSD
  数据存储进程。生产建议每块独立裸数据盘一个 OSD。

Ceph MDS
  CephFS 元数据服务。CephFS 必须有 MDS。

Ceph RGW
  S3 / Swift 对象存储网关。

Ceph CSI
  Kubernetes PVC 动态创建和挂载能力。
```

## 存储能力

### CephFS

CephFS 用于多个 Pod 同时挂载同一个目录：

```yaml
accessModes:
  - ReadWriteMany
storageClassName: rook-cephfs
```

适合：

```text
共享文件目录
共享上传目录
需要 POSIX 文件系统语义的应用
```

不适合：

```text
高性能数据库数据目录
把所有对象存储数据当普通文件目录手工管理
```

### RGW

RGW 提供 S3 兼容 API：

```text
http://rook-ceph-rgw-objectstore.rook-ceph.svc:80
```

适合：

```text
Harbor registry blob
应用附件
备份归档
LFS / package / artifact
大文件分片上传
```

RGW 支持常见 S3 API，包括 multipart upload。它不是 MinIO Console 这种用户控制台；运维管理主要通过 Ceph Dashboard、`radosgw-admin`、`aws cli`、`s3cmd`、`mc` 等工具完成。

## 容量规划

3 台机器，每台 10TB 数据盘，副本数 `size=3`：

```text
原始容量：10TB x 3 = 30TB
理论可用：30TB / 3 = 10TB
建议长期使用：约 7TB
```

如果按 TiB 计算，厂商标称 10TB 约等于 9.1TiB：

```text
理论可用：约 9.1TiB
建议长期使用：约 6.3TiB
```

3 节点不建议用 `size=2` 省容量。节点故障或恢复期间风险会明显变大。

## 故障能力

默认副本数为 3，故障域按 host：

```text
允许 1 台节点故障
不允许 2 台节点同时故障
```

1 台节点故障时：

```text
MON 仍有 2/3 多数派
OSD 仍有 2/3 副本
CephFS MDS 可切换
RGW 可通过剩余实例继续服务
```

恢复节点或替换磁盘后，Ceph 会进行 backfill / rebalance。恢复期间不要继续大规模写入到接近满盘。

## 生产边界

最低建议：

```text
每台节点 CPU：4 核起
每台节点内存：8G 起，16G 更合理
每台节点磁盘：系统盘 + 独立裸数据盘
网络：稳定千兆起，低延迟内网
```

不建议：

```text
OSD 使用系统盘目录
3 台 master 同时跑大量业务、etcd、Ceph、数据库
没有监控告警就投入生产
没有恢复演练就承载关键数据
```

## 本地虚机验证边界

OrbStack、普通 Docker 虚机、单机虚拟化环境可以做 manifest、脚本和小规模功能预检，但不适合作为 Rook Ceph 生产验证环境。

原因：

```text
虚机根盘通常共享宿主机磁盘
loop 文件不等价于独立裸数据盘
containerd 导入和解包 Ceph 镜像会放大磁盘占用
etcd、K3s、containerd、Ceph 同时抢同一块宿主机 IO
```

在 3 master K3s 上验证 Rook Ceph，建议至少准备：

```text
每台节点 1 块真实独立数据盘
每台节点系统盘可用空间 80GiB 以上
每台节点内存 16GiB 更合理
验证环境允许长时间高 IO
```
