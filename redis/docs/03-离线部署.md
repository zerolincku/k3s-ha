# 离线部署

Redis Sentinel 离线部署需要提前准备容器镜像。K3s 本身的离线部署由 `k3s/` 目录描述，本目录只描述 Redis 资源。

## 下载镜像

在有网络的机器拉取固定版本镜像，并导出为 tar：

```bash
cp redis/config.example.env redis/prod.env
vim redis/prod.env

bash redis/scripts/download-redis-images.sh redis/prod.env
```

脚本默认输出：

```text
redis/artifacts/images/redis-7.2.4.tar
```

可以通过配置覆盖：

```bash
REDIS_IMAGE=redis:7.2.4
REDIS_IMAGE_PLATFORM=linux/arm64
REDIS_IMAGE_TAR=/data/offline/redis-7.2.4-arm64.tar
```

## 上传到 K3s 自动导入目录

K3s 会从每个节点的以下目录导入镜像归档：

```text
/var/lib/rancher/k3s/agent/images/
```

配置节点 SSH 地址：

```bash
REDIS_NODE_SSH_HOSTS=k3s-master-1,k3s-master-2,k3s-master-3
REDIS_NODE_NAMES=k3s-master-1,k3s-master-2,k3s-master-3
SSH_USER=root
SSH_PORT=22
REDIS_IMAGE_TAR=/data/offline/redis-7.2.4-arm64.tar
```

上传镜像：

```bash
bash redis/scripts/upload-redis-images.sh redis/prod.env
```

上传脚本会先把镜像归档传成 `.uploading-*.tmp`，传输完成后再 `mv` 为最终 `.tar` 文件。不要手工把正在传输的半成品直接写入最终文件名，否则 K3s 启动扫描 images 目录时可能读到未完成 tar，出现 `short read` 或 `unexpected EOF`。

如果 K3s 是新安装或即将重启，上传到 images 目录即可，K3s 启动时会自动导入。

如果 K3s 已经在运行，只把 tar 放进 images 目录不会立即完成导入。需要重启对应节点的 K3s 服务，才会立刻触发自动导入。生产环境建议逐台滚动重启，确保 etcd 和 API 始终保持可用。

如果 K3s 已经在运行，并且希望立即验证镜像已导入，可以让脚本逐台滚动重启 K3s：

```bash
REDIS_RESTART_K3S_AFTER_IMAGE_UPLOAD=true \
bash redis/scripts/upload-redis-images.sh redis/prod.env
```

如果节点之前已经导入过同名归档，但后来手工删除了 containerd 里的 Redis 镜像，K3s 的 `.cache.json` 可能认为归档未变化而跳过导入。测试或强制重导时可以显式设置：

```bash
REDIS_RESET_K3S_IMAGE_CACHE=true \
REDIS_RESTART_K3S_AFTER_IMAGE_UPLOAD=true \
bash redis/scripts/upload-redis-images.sh redis/prod.env
```

这个操作会删除 K3s images 目录下的导入缓存，下一次 K3s 启动会重新检查并导入归档。生产环境使用前要评估重启窗口。

确认每台节点已导入镜像：

```bash
k3s ctr images ls | grep 'redis:7.2.4'
```

## 使用内网 registry

如果生产环境有内网 registry，建议同步镜像后使用内网地址：

```bash
docker tag redis:7.2.4 registry.example.com/library/redis:7.2.4
docker push registry.example.com/library/redis:7.2.4
```

配置：

```bash
REDIS_IMAGE=registry.example.com/library/redis:7.2.4
```

如果 registry 需要认证，应先按 K3s 或 namespace 的镜像拉取认证方案准备 `imagePullSecret`。当前基础 manifest 默认不内置 registry 凭据。

## 执行部署

离线环境中执行同一个部署脚本：

```bash
bash redis/scripts/deploy-redis-sentinel.sh redis/prod.env
```

部署脚本不会下载任何二进制，只会通过 `kubectl apply` 创建 Kubernetes 资源。只要目标节点已经通过 K3s images 目录导入 Redis 镜像，Pod 就可以离线启动。

## 离线检查

确认 Pod 没有尝试外部拉镜像：

```bash
kubectl -n redis describe pod redis-0 | grep -E 'Image|Pull|Err|BackOff'
```

确认 Sentinel 正常：

```bash
kubectl -n redis exec sentinel-0 -- \
  redis-cli -p 26379 sentinel masters
```
