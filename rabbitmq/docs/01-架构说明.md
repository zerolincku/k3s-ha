# 架构说明

## 推荐拓扑

在 3 master K3s 上部署 3 个 RabbitMQ 节点：

```text
k3s-master-1  -> rabbitmq-server-0
k3s-master-2  -> rabbitmq-server-1
k3s-master-3  -> rabbitmq-server-2
```

每个 RabbitMQ Pod 使用独立 PVC：

```text
rabbitmq-server-0 -> persistence-rabbitmq-server-0
rabbitmq-server-1 -> persistence-rabbitmq-server-1
rabbitmq-server-2 -> persistence-rabbitmq-server-2
```

K3s 默认 `local-path` 存储会为 PV 生成节点亲和。Pod 删除后会回到原节点挂载原本地目录。RabbitMQ 高可用依赖 quorum queue 跨 RabbitMQ 节点复制，不依赖共享存储。

## 为什么不用共享存储

RabbitMQ 节点不应该多个 Pod 挂同一块数据盘。RabbitMQ 每个节点维护自己的数据目录、Mnesia 元数据和队列数据。共享存储不能替代 RabbitMQ 集群复制，反而可能放大数据损坏和调度风险。

本方案使用：

```text
独立 PVC + local-path PV nodeAffinity + podAntiAffinity + quorum queue
```

职责划分：

- PVC 负责单个 RabbitMQ 节点的数据保留。
- local-path PV nodeAffinity 负责 Pod 回到本地数据所在节点。
- podAntiAffinity 负责首次调度尽量分散到 3 台节点。
- quorum queue 负责消息跨 RabbitMQ 节点复制。

## 队列高可用

生产需要高可用和数据安全的队列应使用 quorum queue：

```text
x-queue-type=quorum
```

本方案在 RabbitMQ 配置里设置：

```text
default_queue_type = quorum
```

仍然建议业务显式声明队列类型，避免不同客户端或历史队列行为不一致。

不建议使用 classic mirrored queue。它是过时方案，在 RabbitMQ 4.0 起已移除。

## 故障模型

3 节点 RabbitMQ quorum queue 的容错边界：

```text
故障 1 个 RabbitMQ 节点：多数派仍在，队列可用。
故障 2 个 RabbitMQ 节点：多数派丢失，相关 quorum queue 不可用。
```

这是正确行为，用于避免脑裂和消息不一致。

## 服务入口

Operator 会创建客户端 Service：

```text
rabbitmq.rabbitmq.svc.cluster.local:5672
rabbitmq.rabbitmq.svc.cluster.local:15672
```

集群内业务使用 ClusterIP 即可。集群外业务访问 AMQP `5672` 时，应通过 TCP 负载均衡、NodePort 或后续统一入口暴露，不建议直连某个 Pod。

RabbitMQ 管理控制台 `15672` 是 HTTP 服务，可以通过 Ingress 暴露。AMQP `5672` 是 TCP 协议，普通 HTTP Ingress 不能直接代理；需要使用 NodePort、LoadBalancer、HAProxy TCP 转发，或者支持 TCP stream 的 Ingress Controller。

## 业务侧必须配合

RabbitMQ 服务端高可用不能单独保证业务不丢消息。业务侧需要：

- durable exchange
- durable quorum queue
- persistent message
- publisher confirm
- consumer ack
- 合理的重试和死信策略
