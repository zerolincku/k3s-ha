# 离线部署方案

离线部署分两段：在有网络的机器准备离线包，在离线环境执行同一个部署脚本。

## 离线包内容

离线包包含：

```text
k3s
install.sh
k3s-airgap-images-amd64.tar.zst
sha256sum-amd64.txt
metadata.env
```

`arm64` 离线包中的文件名对应为：

```text
k3s
install.sh
k3s-airgap-images-arm64.tar.zst
sha256sum-arm64.txt
metadata.env
```

说明：离线包内部统一把 K3s binary 放成 `k3s`，但下载时 `amd64` 使用 release 里的 `k3s`，`arm64` 使用 release 里的 `k3s-arm64`。

如果目标系统启用 SELinux，还需要单独准备：

```text
k3s-selinux-*.rpm
container-selinux
policycoreutils
selinux-policy
```

OS 包 `curl`、`socat`、`conntrack-tools` 等不包含在 K3s 官方离线包内，需要提前通过内网 yum/apt 源或 OS 安装包仓库解决。Keepalived/HAProxy 的离线依赖由 `keepalived-haproxy/` 目录单独描述。

## 准备离线包

在有网络的机器执行：

```bash
cp k3s/配置示例.env k3s/prod.env
vim k3s/prod.env

bash k3s/scripts/prepare-airgap-bundle.sh k3s/prod.env
```

默认输出：

```text
k3s/artifacts/airgap/k3s-airgap-bundle-<version>-amd64.tar.gz
```

可以通过环境变量指定版本和架构：

```bash
K3S_VERSION=v1.34.8+k3s1 \
K3S_ARCH=amd64 \
bash k3s/scripts/prepare-airgap-bundle.sh k3s/prod.env
```

命令行环境变量会覆盖 `k3s/prod.env` 中的同名配置，适合临时生成某个版本或某个架构的离线包。

同时准备 `amd64` 与 `arm64`：

```bash
K3S_VERSION=v1.34.8+k3s1 \
K3S_ARCH=all \
bash k3s/scripts/prepare-airgap-bundle.sh k3s/prod.env
```

输出示例：

```text
k3s/artifacts/airgap/k3s-airgap-bundle-v1.34.8+k3s1-amd64.tar.gz
k3s/artifacts/airgap/k3s-airgap-bundle-v1.34.8+k3s1-arm64.tar.gz
```

## 离线部署

把离线包复制到运维机，然后执行：

```bash
K3S_AIRGAP=true \
AIRGAP_BUNDLE_AMD64=/path/to/k3s-airgap-bundle-v1.34.8+k3s1-amd64.tar.gz \
AIRGAP_BUNDLE_ARM64=/path/to/k3s-airgap-bundle-v1.34.8+k3s1-arm64.tar.gz \
bash k3s/scripts/deploy-k3s-ha.sh k3s/prod.env
```

即使 `k3s/prod.env` 中保留了 `K3S_AIRGAP=false`，上面的命令行环境变量也会覆盖它。

如果三台机器都是同一架构，也可以只设置：

```bash
AIRGAP_BUNDLE=/path/to/k3s-airgap-bundle-v1.34.8+k3s1-amd64.tar.gz
```

脚本会将离线包分发到每台 master，并在每台机器上：

1. 按节点架构选择对应离线包。
2. 解压 `k3s` 到 `/usr/local/bin/k3s`。
3. 解压 `install.sh` 到 `/opt/k3s-airgap/install.sh`。
4. 复制 `k3s-airgap-images-<arch>.tar.zst` 到 `/var/lib/rancher/k3s/agent/images/`。
5. 创建 `.cache.json`，避免 K3s 重启时重复导入未变化的镜像归档。
6. 使用 `INSTALL_K3S_SKIP_DOWNLOAD=true` 执行本地安装脚本。

## 离线镜像策略

小规模 3 master 集群有两种选择：

| 方式 | 适用场景 | 说明 |
| --- | --- | --- |
| 每节点放置 airgap tarball | 最简单，适合 3 台机器 | 本方案默认使用 |
| 内网私有 registry | 后续服务很多，镜像治理要求高 | 需要维护 `registries.yaml` 和镜像同步流程 |

后续 Redis、RabbitMQ、MinIO 等服务建议统一进入内网 registry。K3s 自身可继续用 airgap tarball，也可切换为 registry 方式。

混合架构离线集群强烈建议使用支持 multi-arch manifest 的内网 registry。单独导入某个架构镜像 tarball 只能保证该节点本地可用，不能保证业务 Pod 在另一种架构节点上也能拉起。

如果在线部署时发现 `kube-system` Pod 卡在 `ImagePullBackOff`，并且事件中出现 `registry-1.docker.io`、`auth.docker.io`、`EOF` 或 `tls: handshake failure`，不要继续依赖公网重试。应改用本章的 airgap 包，或先把 K3s 系统镜像完整同步到内网 registry。

## 升级流程

1. 在联网机器下载新版本 K3s binary、install.sh、airgap images。
2. 生成新的离线包并校验 checksum。
3. 先备份 etcd snapshot。
4. 逐台 master 执行升级，保证任意时刻至少 2 台 master 健康。
5. 验证 kube-system Pod、节点版本、业务入口。

升级前备份：

```bash
sudo k3s etcd-snapshot save --name pre-upgrade-$(date +%Y%m%d%H%M%S)
```
