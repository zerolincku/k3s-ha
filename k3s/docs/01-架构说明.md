# K3s 3 Master 高可用架构

## 目标

在仅有 3 台服务器的前提下，部署可用于生产的 K3s 高可用控制面：

- 3 台机器全部作为 K3s server/master。
- 使用 K3s embedded etcd，避免额外维护外部数据库。
- 不要求先部署 VIP/HAProxy；master2、master3 默认通过 master1 的 `6443` 加入。
- 后续可以单独增加 API 入口，也可以单独设计业务对外暴露。

## 推荐拓扑

| 角色 | 示例 IP | 组件 |
| --- | --- | --- |
| master1 | `192.168.1.11` | k3s server、embedded etcd |
| master2 | `192.168.1.12` | k3s server、embedded etcd |
| master3 | `192.168.1.13` | k3s server、embedded etcd |
| optional API endpoint | `192.168.1.10:8443` | 可后续由 `keepalived-haproxy/` 或外部 LB 提供 |

请求路径：

```text
master2 / master3 join
  -> https://master1:6443

kubectl / automation after optional API LB
  -> https://VIP:8443
  -> healthy k3s server:6443
```

## 为什么是 3 master + embedded etcd

3 台机器场景下，推荐直接部署 3 个 K3s server：

- `1 master + 2 worker` 不是高可用控制面。master 故障后，已有 worker 上的 Pod 可能继续运行，但调度、重建、扩缩容、Service 变更等控制面能力不可用；如果工作负载正好在 master 上，也会一起丢失。
- `2 master + 1 worker` 不适合 embedded etcd。etcd 两成员集群无法容忍任意 1 个成员故障，失去一个成员后无法保持 quorum。
- `3 master` 是 3 台机器下最小可用的 etcd quorum 形态，可容忍 1 台 master 故障。

## 端口设计

| 端口 | 方向 | 用途 |
| --- | --- | --- |
| `6443/tcp` | client / other masters -> master | K3s apiserver |
| `8443/tcp` | client -> optional API endpoint | 可选 API LB 对外入口 |
| `2379-2380/tcp` | master <-> master | embedded etcd |
| `8472/udp` | node <-> node | Flannel VXLAN，使用默认 CNI 时需要 |
| `10250/tcp` | control plane -> node | kubelet |

如果后续 API LB 与 K3s colocated 在同一批 master 节点，外部入口建议使用 `8443`，避免 LB 与 K3s apiserver 争用本机 `6443`。具体 Keepalived/HAProxy 配置放在 `keepalived-haproxy/` 目录。

## 生产前置条件

- 3 台机器时间同步，建议启用 chrony。
- 主机名唯一，静态 IP 固定，DNS 或 `/etc/hosts` 可解析。
- 三台机器之间放通上述端口。
- 明确每台机器 CPU 架构：`x86_64` 映射为 `amd64`，`aarch64/arm64` 映射为 `arm64`。
- 明确每台机器 cgroup 模式：cgroup v1 与 cgroup v2 都可以作为普通 rootful K3s 节点使用，但生产环境建议同一批 master 保持一致，降低排障复杂度。
- 禁止 swap，或确认 kubelet 配置允许当前策略。
- 内核参数允许转发与桥接流量：

```text
net.ipv4.ip_forward = 1
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1
```

- 生产环境必须固定 K3s 版本，不使用浮动 latest。
- 如果 SELinux 开启，离线环境要提前准备 `k3s-selinux` 及其依赖。

## 架构与 cgroup 策略

### CPU 架构

K3s 官方 release 针对不同架构提供不同资源：

| 架构 | 主机 `uname -m` | K3s binary | airgap images |
| --- | --- | --- | --- |
| `amd64` | `x86_64` | `k3s` | `k3s-airgap-images-amd64.tar.zst` |
| `arm64` | `aarch64` / `arm64` | `k3s-arm64` | `k3s-airgap-images-arm64.tar.zst` |

在线部署时脚本会识别每台 master 的架构，K3s install script 会下载匹配二进制。离线部署时，必须为所有目标架构准备对应离线包。

混合架构集群可以运行，但后续 Redis、RabbitMQ、MinIO 等业务镜像必须支持 multi-arch manifest，或者显式使用 `nodeSelector` / `affinity` 把单架构镜像调度到匹配节点。

### cgroup v1 / v2

脚本会在每台 master 上检测：

```bash
test "$(stat -fc %T /sys/fs/cgroup)" = "cgroup2fs" && echo v2 || echo v1
```

默认配置：

```bash
CGROUP_MODE=auto
K3S_CGROUP_DRIVER=auto
```

含义：

- `CGROUP_MODE=auto`：只检测并打印，不要求三台机器一致。
- `CGROUP_MODE=v1` 或 `v2`：要求所有 master 与指定模式一致，不一致则中止部署。
- `K3S_CGROUP_DRIVER=auto`：保留 K3s/containerd 默认匹配逻辑。
- `K3S_CGROUP_DRIVER=systemd|cgroupfs`：显式传给 kubelet，仅在已经验证目标 OS/containerd 基线时使用。

生产建议：同一集群 master 尽量统一 OS 大版本、内核、cgroup 模式和 containerd 配置。混用 cgroup v1/v2 虽然可行，但后续排查资源限制、驱逐、监控指标时成本更高。

## K3s 关键配置

所有 master 使用一致的 cluster token。若已经确定后续 API VIP 或域名，可以提前写入 TLS SAN；如果暂时没有，也可以不写：

```yaml
token: "<shared-token>"
tls-san:
  - "192.168.1.10"
write-kubeconfig-mode: "0600"
disable:
  - traefik
kubelet-arg:
  - "node-labels=node.k3s-ha.io/cgroup-version=v2"
```

第一个 master 初始化 embedded etcd：

```yaml
cluster-init: true
```

后两个 master 默认通过 master1 加入：

```yaml
server: "https://192.168.1.11:6443"
```

## API 入口与业务入口的边界

K3s 控制面部署、Kubernetes API 对外入口、业务服务对外暴露是三件事：

| 层次 | 作用 | 可选方案 |
| --- | --- | --- |
| K3s 控制面 | 运行 apiserver、scheduler、controller、embedded etcd | 本目录完成 |
| Kubernetes API 入口 | 给 kubectl、自动化、后续节点加入提供统一入口 | master1:6443、Keepalived+HAProxy、外部 LB |
| 业务服务入口 | 对外暴露应用流量 | NodePort、Ingress、LoadBalancer/MetalLB |

所以可以先部署 K3s，再单独做 VIP/HAProxy；业务服务也不一定依赖 API VIP，可以按业务需要选择 NodePort 或 Ingress。

## 故障模型

| 故障 | 预期影响 | 处理 |
| --- | --- | --- |
| 单 master 故障 | etcd 仍有 2/3 quorum，控制面可用 | 修复节点后重新加入或恢复 |
| API LB 故障 | K3s server 仍可运行，但统一 API 入口受影响 | 按 `keepalived-haproxy/` 或外部 LB 手册检查 |
| 两台 master 故障 | etcd 丢失 quorum，控制面不可用；已有 Pod 不一定立即退出 | 先恢复至少 2 个 etcd 成员 |
| 可选 VIP 冲突 | API 入口不稳定 | 确认同网段唯一 VIP，关闭其他占用 |

## 后续组件落位

K3s 底座完成后，中间件建议优先补齐：

1. 默认 StorageClass 与数据盘策略。
2. Ingress 与证书体系。
3. 监控告警。
4. Redis、RabbitMQ、MinIO 等有状态服务高可用方案。
5. Etcd snapshot 与对象存储备份。
