# 运维手册

## 常用检查

RabbitMQ 资源：

```bash
kubectl -n rabbitmq get rabbitmqcluster
kubectl -n rabbitmq get pods -o wide
kubectl -n rabbitmq get pvc
kubectl -n rabbitmq get svc
```

Operator 状态：

```bash
kubectl -n rabbitmq-system get pods
kubectl -n rabbitmq-system logs deploy/rabbitmq-cluster-operator
```

RabbitMQ 集群状态：

```bash
kubectl -n rabbitmq exec rabbitmq-server-0 -- rabbitmq-diagnostics cluster_status
kubectl -n rabbitmq exec rabbitmq-server-0 -- rabbitmq-diagnostics status
```

队列状态：

```bash
kubectl -n rabbitmq exec rabbitmq-server-0 -- rabbitmqctl list_queues name type durable messages consumers
kubectl -n rabbitmq exec rabbitmq-server-0 -- rabbitmq-queues quorum_status --all
```

## 存储和节点亲和检查

查看 Pod 分布：

```bash
kubectl -n rabbitmq get pods -o wide
```

查看 PVC：

```bash
kubectl -n rabbitmq get pvc
```

查看 PV 的 nodeAffinity：

```bash
for pvc in $(kubectl -n rabbitmq get pvc -o name); do
  pv=$(kubectl -n rabbitmq get "$pvc" -o jsonpath='{.spec.volumeName}')
  echo "===== $pvc -> $pv ====="
  kubectl get pv "$pv" -o yaml | grep -A20 nodeAffinity
done
```

如果使用 K3s `local-path`，PV 会绑定到第一次创建数据目录的节点。删除 Pod 后，Pod 应回到原节点。

## 故障验证

删除一个 RabbitMQ Pod：

```bash
kubectl -n rabbitmq delete pod rabbitmq-server-0
```

等待恢复：

```bash
kubectl -n rabbitmq rollout status statefulset/rabbitmq-server --timeout=600s
```

检查 quorum：

```bash
kubectl -n rabbitmq exec rabbitmq-server-0 -- rabbitmq-queues quorum_status --all
```

3 节点 quorum queue 只能容忍 1 个 RabbitMQ 节点故障。故障 2 个节点时，队列会失去多数派并不可用。

## 客户端要求

业务侧必须配合：

```text
durable exchange
durable quorum queue
persistent message
publisher confirm
consumer ack
```

如果没有 publisher confirm，生产者无法确认消息是否真正进入 RabbitMQ。如果没有 consumer ack，消费者异常时可能误认为消息已处理。

## 卸载

删除 RabbitMQ 集群：

```bash
kubectl delete namespace rabbitmq
```

删除 Operator：

```bash
kubectl delete -f rabbitmq/artifacts/operator/cluster-operator-v2.21.0.yml
```

如果使用的是在线安装：

```bash
kubectl delete -f https://github.com/rabbitmq/cluster-operator/releases/download/v2.21.0/cluster-operator.yml
```

生产环境删除前必须确认消息已消费或已迁移。
